## 3. 專案名稱

`臺灣宜居地智慧分析平台：基於環境資料與AI分析的城市宜居度評估系統`

中文名稱建議：

`Taiwan Livability AI Platform`

## 4. 專案摘要

本專案建立一個結合 GIS、開放資料分析與生成式 AI 的決策平台，讓使用者能從地圖與自然語言兩種方式理解某區域的宜居性、風險暴露程度與生活機能狀況，並得到可解釋的建議。

平台核心價值：

- 將分散的開放資料整合為單一決策視圖。
- 用 AI 將專業資料轉成一般人可理解的語句。
- 協助使用者快速比較不同區域，找出更適合居住、就學、投資或設點的位置。

## 5. 問題定義

一般民眾、租屋族、購屋族、學生、創業者或中小企業，在評估某區域是否適合生活或設點時，常面臨以下問題：

- 公開資料分散，難以跨來源整合。
- 地理資訊難讀，非專業使用者不易解讀。
- 風險資料與生活機能資料常被分開看，缺乏整體判讀。
- 即使資料存在，也難以快速轉換成行動建議。

因此，本專案要解決的核心問題是：

`如何把地理空間資料、環境風險資料與生活便利性資料整合成可查詢、可比較、可解釋、可視覺化的 AI 決策工具。`

## 6. 目標使用者

- 租屋族與購屋族
- 學生與家長
- 小型店家與創業團隊
- 地方治理與社區規劃單位
- 競賽評審與一般展示觀眾

## 7. 專案目標

### 7.1 商業與展示目標

- 在 3 到 5 分鐘 Demo 內，清楚展示資料整合、地圖分析與 AI 問答能力。
- 呈現明確使用情境，而非只展示模型或聊天機器人。
- 具備延伸成 SaaS、B2G 或 B2B2C 服務的潛力。

### 7.2 技術目標

- 建立可重複執行的資料清理與整併流程。
- 提供地圖查詢、區域評分、條件篩選與自然語言問答。
- 產出具可解釋性的「區域評估結果」。
- 支援至少一個可操作的前端介面。

### 7.3 MVP 成功條件

- 使用者可選擇行政區、地址或地圖點位。
- 系統可回傳該區域的宜居分數與構成原因。
- 系統可顯示風險圖層與生活機能圖層。
- 使用者可用自然語言詢問區域分析結果。
- 系統可輸出簡短建議，例如「適合學生租屋」「不建議設置親子空間」。

## 9. 使用情境

### 9.1 租屋族

使用者輸入「台中西屯區哪裡比較適合學生租屋？」系統回傳：

- 地圖熱區
- 各候選區域分數
- 附近機能與風險說明
- 綜合建議

### 9.2 家長

使用者點選某個區域，查看：

- 空氣品質
- 災害風險
- 交通便利度
- 醫療與學校可近性

### 9.3 創業者

使用者詢問「我想開咖啡店，哪些地段生活機能高但環境風險低？」系統提供排序與解釋。

## 10. 功能需求

### 10.1 地圖互動

- 顯示底圖與行政區圖層
- 支援縮放、點選、圖層切換
- 點選區域後顯示指標摘要

### 10.2 區域評分

- 將多來源指標標準化
- 依權重計算綜合宜居分數
- 顯示各分項得分

分項建議：

- 環境品質
- 生活機能
- 交通便利
- 公共服務可近性
- 災害或風險暴露

### 10.3 條件篩選

- 可依使用者偏好調整權重
- 可篩選高分區域
- 可比較 2 至 3 個區域

### 10.4 AI 問答

- 支援自然語言輸入
- 將問題轉成資料查詢與分析流程
- 回傳結論、依據與簡短建議

### 10.5 分析摘要

- 產出一段 100 到 250 字的摘要
- 明確說明優勢、風險與適用情境

### 10.6 匯出功能

- 匯出單區分析結果為圖片或 PDF
- 匯出比較表格為 CSV

## 11. 非功能需求

- 回應時間：一般查詢 3 秒內，AI 問答 10 秒內
- 可用性：Demo 過程中不得依賴人工修正資料
- 可維護性：資料處理、模型、前端需分層
- 可解釋性：每個評分都要有明確來源與計算方式
- 穩定性：即使 AI 失敗，也必須保留基本查詢與地圖展示

## 12. 系統架構

建議採三層架構：

### 12.1 Data Layer

- 蒐集開放資料
- 清理、轉換、座標統一
- 輸出 GeoJSON、CSV 或 Parquet

### 12.2 Service Layer

- 評分引擎
- GIS 分析模組
- AI 問答服務
- API 封裝

### 12.3 Presentation Layer

- Web 前端
- 地圖互動 UI
- 問答介面
- 圖表與說明卡片

## 13. 技術選型

### 13.1 建議技術棧

- 語言：Python 3.11+
- 後端：FastAPI
- 前端：Streamlit 或 React
- GIS：GeoPandas、Shapely、Folium 或 Mapbox
- 數據處理：Pandas、NumPy
- 模型：scikit-learn、XGBoost
- LLM 介面：OpenAI API 或本地/開源模型封裝
- 向量資料庫：Chroma 或 FAISS

### 13.2 MVP 建議簡化方案

若時間有限，優先採：

- 前端：Streamlit
- 後端：直接在 Streamlit 內包服務邏輯
- 分析：規則式評分 + 可解釋權重
- AI：以 API 模型做摘要與問答，不先訓練大型模型

這樣能在最短時間內產出可展示成果。

## 14. 資料需求

### 14.1 核心資料類型

- 空氣品質資料
- 人口或生活機能資料
- 醫療、學校、交通節點 POI
- 災害潛勢或環境風險資料
- 行政區界資料

### 14.2 資料處理原則

- 所有資料要統一座標系統
- 明確記錄更新日期
- 缺值要有處理策略
- 每個欄位需有中文說明

### 14.3 指標欄位建議

每個分析單元至少包含：

- `region_id`
- `region_name`
- `air_quality_score`
- `transport_score`
- `facility_score`
- `risk_score`
- `livability_score`
- `summary_text`

## 15. 分析與評分邏輯

### 15.1 分析單位

建議先用以下其中一種：

- 行政區
- 里
- 固定網格

MVP 建議使用行政區，因為資料整合成本最低。

### 15.2 評分方式

建議先採加權指標法，不必一開始就做預測模型。

範例：

`livability_score = 0.25 * air + 0.25 * facility + 0.2 * transport + 0.15 * public_service + 0.15 * (100 - risk)`

### 15.3 模型策略

分兩階段：

- 第一階段：規則式評分與標準化
- 第二階段：若資料量足夠，再加入回歸或排序模型優化權重

這樣較符合競賽節奏，也比較容易向評審解釋。

## 16. AI 問答設計

### 16.1 目標

讓使用者不需要懂 GIS 或資料欄位，也能直接問：

- 哪裡適合學生租屋？
- 哪區生活機能高又相對安全？
- 某區不適合親子活動的主因是什麼？

### 16.2 問答流程

1. 使用者輸入自然語言問題
2. 系統辨識意圖、區域與偏好條件
3. 查詢對應資料與評分結果
4. 產生結論、理由與建議
5. 視需要高亮對應地圖區域

### 16.3 安全與限制

- LLM 不直接虛構資料來源
- 每段結論需綁定真實數據欄位
- 若查無資料，必須明確回覆

## 17. API 規格建議

### 17.1 `GET /regions`

回傳所有可查詢區域清單。

### 17.2 `GET /region/{id}`

回傳單一區域的詳細分數、原始指標與摘要。

### 17.3 `POST /score`

輸入使用者權重，回傳重新計算後的排序結果。

### 17.4 `POST /ask`

輸入自然語言問題，回傳：

- 分析摘要
- 涉及區域
- 關鍵指標
- 推薦或不推薦原因

## 24. 專案風險

- 開放資料格式不一致，清理成本高
- GIS 資料座標系不一致
- AI 回答容易失真，若沒做資料綁定會被評審質疑
- 若功能做太多，容易導致 Demo 不穩

對策：

- 先做單一城市
- 先做規則式可解釋評分
- AI 僅負責摘要與問答，不負責核心計算
- 每個頁面都設計 fallback 畫面

## 25. 建議開發順序

你們接下來不要先做聊天機器人，正確順序應該是：

1. 先確認要示範哪個城市
2. 先確定 3 到 5 個資料來源
3. 先把區域主表與評分算出來
4. 先做可操作地圖與分數卡
5. 最後再串 AI 問答

原因很直接：沒有穩定資料底座，AI 只會變成空殼展示。
